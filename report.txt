PLOSHA-RMFR — Session Report
Debugging run_exp1.sh / run_exp2.sh on AWS, statistics correction, and
investigation of an aggregation-latency anomaly in Experiment 1.
Last updated: 2026-07-22


===============================================================================
1. WHAT WENT WRONG, ORIGINALLY
===============================================================================

run_exp1.sh failed to run at all on the AWS instance (Ubuntu, x86-64).
Root causes, found in order:

  a) run_exp1.sh had CRLF line endings (edited/saved on Windows). Bash on
     Linux read the trailing \r as part of each line, producing errors like:
         run_exp1.sh: line 2: set: -: invalid option
         run_exp1.sh: line 3: $'\r': command not found
         cd: $'/path\r/schemes/...': No such file or directory

  b) The committed binary schemes/plosha_rmfr/src/plosha_rmfr was a macOS
     arm64 Mach-O executable (built on a Mac and checked into git). Linux
     cannot execute a Mach-O image at all — this is a different binary
     format, not a missing-library problem.

  c) The AWS instance had no C++ toolchain and no OpenSSL headers installed
     (needed by every scheme's Makefile: -lssl -lcrypto -pthread).

  d) run_exp1.sh itself was untracked in git and briefly went missing from
     the working directory entirely; it had to be reconstructed.

  e) 14 compiled binaries across all 5 schemes were committed to git with no
     .gitignore rule to prevent it (only *.o / *.exe were ignored, not
     extensionless ELF/Mach-O executables).

3. STATISTICS FIX (mean/std estimator)
===============================================================================

Found: recent commits had changed Experiment 1's aggregation ablation
(runExp8_AblationAggregation, des_engine.cpp) to summarise each sweep point
by MEDIAN instead of mean, using a bespoke lambda written inline — a THIRD
implementation of "summarise these epochs," alongside the one already used by
every other experiment (MetricsCollector::computeAverages, mean + population
std). Worse, the "std_*" columns in that code were left computing deviation
about the MEDIAN while being labelled and reported as if they were standard
deviations — they were not: they systematically overstate real dispersion,
and two of the six metrics (recomputation_overhead, reused_microslot_count)
were still centered on the MEAN in the same block, so the file was internally
inconsistent.

Fairness check: all four ablation variants (flat_epoch, fixed_slot,
adaptive_slot, full_plosha) go through the identical code path and identical
estimator, so switching estimators did not advantage any one variant over
another — it was not a p-hacking risk, but it was still wrong.

Fix applied:
  - Removed the ~58-line inline median/variance block from des_engine.cpp.
  - Extended the existing MetricsCollector::computeAverages() (used by every
    other experiment) to also compute mean + population std for the six
    ablation metrics.
  - runExp8 now calls this one shared function, same as every other
    experiment. Net effect: -74 lines in des_engine.cpp, one estimator used
    everywhere, std_* columns are now genuine standard deviations.

Verified: compiled clean under GCC 15 (0 errors), and cross-checked that the
mean reported in results.csv for a given sweep point exactly matches the mean
computed independently from that point's raw per-epoch dump (see section 5) —
5.268775 in both, to 6 decimal places.


===============================================================================
4. --epochs AND --failure-rate WERE SILENTLY IGNORED
===============================================================================

Found while trying to run a 2-epoch smoke test of run_exp1.sh: passing
EPOCHS=2 still ran the full 30 epochs. Cause: runExp8_AblationAggregation
(and, it turned out, six other experiment runners — Exp1 through Exp6)
overwrote exp_config.num_epochs and/or exp_config.failure_rate with hardcoded
values AFTER copying the CLI-supplied config, silently discarding whatever
the user passed on the command line.

Fix:
  - Added ExperimentConfig::failure_rate_set (set by the CLI parser only
    when --failure-rate is actually supplied), so an experiment with its own
    documented default can tell "caller asked for X" apart from "caller left
    it unset" instead of unconditionally overwriting it.
  - Added a named constant EXP1_ABLATION_FAILURE_RATE = 0.50 (config.hpp)
    documenting why Experiment 1 defaults higher than the global 0.05 default
    (recovery cost is what separates the ablation variants; too few epochs
    trigger recovery at 5%) and flagging that it lies outside the 0.02-0.35
    range swept by the failure-rate experiment.
  - Removed six hardcoded `exp_config.num_epochs = 30/20` overwrites across
    Exp1-Exp6 that were pure flag-killers (config.num_epochs already
    defaults to 30/ABLATION_EPOCHS; the overwrite did nothing except break
    --epochs).
  - Exp7 (AFLTO ablation, fixed 20-entry failure/workload schedule arrays)
    and Exp9 (scheduling efficiency, burst at epoch 12 / degradation at
    epoch 21) intentionally KEEP a fixed epoch count — their internal
    per-epoch schedules are indexed by epoch number and would silently
    shift under a different count. This is documented in code comments and
    in each experiment's provenance output (section 5), not silently
    overridden.
  - Every experiment now prints/records the parameters actually in effect,
    so a log or output directory can be matched back to the CLI invocation
    that produced it.

Verified on the AWS box: --epochs 3 now produces exactly 3-epoch runs and
3-row per-epoch dumps; --failure-rate 0.10 is honored and reported as
"(--failure-rate)" rather than "(default)" in both the console output and
run_metadata.txt.


===============================================================================
5. PROVENANCE AND AUDIT-TRAIL ADDITIONS
===============================================================================

Two gaps were identified and closed:

  a) No result file recorded what produced it — not the code version, not
     the host, not whether it was a native or Gramine-SGX run, not the
     effective parameters. Combined with run_benchmark.sh's convention of
     overwriting the plain experiment directories with SGX output after
     backing up native results to a "_native" suffix, native and SGX numbers
     could end up silently mixed with no artifact revealing which was which.

     Fix: MetricsCollector::writeRunMetadata() writes a run_metadata.txt next
     to every results.csv, recording: experiment name, UTC timestamp, git
     commit (via a GIT_COMMIT define injected by the Makefile from
     `git rev-parse`, with a "-dirty" suffix on uncommitted changes),
     hostname, platform (uname), execution_mode (native vs gramine-sgx,
     auto-detected via presence of /dev/attestation, which Gramine mounts
     inside the enclave), effective epochs/failure_rate (and whether the
     latter came from --failure-rate or a default), dataset path, and
     Paillier key size. Wired into all four experiment output paths (the
     generic sweep runner, and Exp7/Exp8/Exp9 individually, since each has
     its own parameter-setting logic).

  b) Per-epoch data was discarded (MetricsCollector::reset() between sweep
     points) — only the aggregated mean/std survived, so neither a reviewer
     nor we could audit whether a given mean/std was well-behaved.

     Fix: MetricsCollector::writeEpochRecordsFile() dumps every recorded
     epoch (all 16 metrics) as its own CSV row, written just before reset()
     destroys the data. One file per sweep point, under
     <experiment_dir>/epoch_records/. Cross-checked against a live run: the
     mean of a dumped epoch file matched the corresponding results.csv value
     exactly.

Status: this code is implemented, synced to the AWS box, and compile-verified
via a scratch binary (0 errors) and a functional test (produced correct
run_metadata.txt and epoch_records/*.csv, confirmed against Experiment 4 as
a test case). It was NOT yet compiled into the production plosha_rmfr binary
used for the Experiment 1 run described in section 6 below — that run
predates these additions and therefore has no run_metadata.txt or
epoch_records/ of its own. It WAS picked up for the Experiment 2 run
(section 7), since that build happened after this code was in place.

7. GRAPH 1 ANOMALY — INVESTIGATION
===============================================================================

Observation: the plot shows a bump in aggregation_latency_ms at 1000 sensors
and a smaller one at 4500 sensors, present in all four variants at the same
x-value.

Initial (WRONG) assessment: I first estimated this as being within one
standard error assuming n=30 samples per point and dismissed it as noise.
This was incorrect — the sweep actually pools 30 independent iterations x 30
epochs = 900 samples per point (metrics_.reset() is called once per sweep
point, not per iteration). At n=900, standard error is ~0.07 ms, and the
observed bumps (0.13-0.37 ms) are 3-5x that — too large, at this sample size,
to dismiss as pure sampling noise. This was corrected once the loop structure
was re-examined.

Given the bump is real (not attributable to sampling noise) but must not be
"fixed" by curve-fitting, smoothing, or filtering data points — that would be
indefensible result-shaping — three structural (non-random) hypotheses were
tested by direct calculation, without rerunning the simulation:

  HYPOTHESIS 1 — Failure-count quantization.
  Failures are injected as num_failures = ceil(weibull_rate * num_fog),
  capped at num_fog-1 (des_engine.cpp). ceil() rounds up, and the rounding
  error is proportionally larger at small num_fog. A standalone C++ program
  replaying just this conversion (200,000 trials per num_fog value) confirmed
  a REAL bias: effective failure fraction ranges from +0.7 percentage points
  (num_fog=5) to -5.4 points (num_fog=50) relative to the nominal 0.50 rate.
  However, this bias is SMOOTH and MONOTONIC across the sweep — it does not
  spike or dip specifically at num_fog=10 (1000 sensors) or num_fog=45
  (4500 sensors). RULED OUT as the cause of the isolated bumps, though the
  underlying bias is a real, independent defect worth fixing (see section 8).

  HYPOTHESIS 2 — Dataset periodicity / sensor ID wraparound.
  The dataset has exactly 100 unique real sensors (dataset_loader.cpp uses
  original_sid = virtual_sid % 100 to cycle through them for larger sensor
  counts). Every value in the sensor sweep (500, 1000, ..., 5000) is an exact
  multiple of 100, so every real sensor profile is reused an equal, whole
  number of times at every single sweep point with no partial-cycle
  remainder. RULED OUT — this mechanism cannot produce a point-specific
  effect under this sweep, by construction.

  HYPOTHESIS 3 — Round-robin sensor-to-fog assignment clustering.
  Sensor-to-fog assignment is deterministic round-robin (sid % num_fog_nodes,
  dataset_loader.cpp). Since real per-sensor heterogeneity does exist (a
  direct read of the CSV found mean vibration ranging 28.99-43.96 across the
  100 real sensors, global std ~2.97), an uneven mapping of "harder" sensors
  onto fewer fog nodes at specific num_fog values could plausibly inflate
  system cost. A second standalone program computed, for each sweep point,
  the standard deviation of per-fog-node average vibration (a direct
  workload-imbalance measure) under the exact assignment used by the
  simulation. Result: imbalance does NOT track the observed bumps —
  4500 sensors (num_fog=45) has one of the LOWEST imbalance values in the
  entire sweep (2.52%), while 5000 sensors (num_fog=50) has the HIGHEST
  (6.09%) yet shows LOWER latency than 4500, the opposite of what this
  hypothesis predicts. 1000 sensors' imbalance (3.42%) is tied exactly with
  3000 sensors, which shows no bump at all. RULED OUT.

CURRENT STATUS: three clean, cheaply-testable structural hypotheses have been
disproven by direct calculation, with no simulation reruns required for any
of them. The bump is real (confirmed via correct sample-size accounting) but
its precise mechanism is not resolved. The most likely remaining explanation
is an interaction with RMFR's recovery-mode threshold logic (tau_1/tau_2/
tau_3, completeness flags) — e.g. certain num_fog values may push a
disproportionate number of epochs across a threshold boundary into a more
expensive recovery path. Confirming this would require runtime
instrumentation (logging which RecoveryMode fires per epoch, per sweep
point) and an actual rerun, which has NOT been done — this is a real cost,
not a cheap static check like the three above.

No further reruns or code changes for this investigation have been made
pending a decision on whether to pursue the threshold-logic hypothesis or
report the bump as-is (real, present in this random realization, with error
bars communicating its magnitude honestly rather than hiding or smoothing it).
